# RefSense Design Notes

## 🖼️ PDF Page Rendering and Image Extraction

### 📌 아이디어 개요

스캔 기반 PDF는 보통 텍스트 레이어가 없으며, 주요 콘텐츠는 단일 이미지(XObject)로 구성됨. 따라서 PDF 전체를 렌더링하는 대신, 해당 이미지를 추출하여 활용하는 방식이 효율적일 수 있음.

### ✅ 관찰된 특성

* 스캔된 PDF는 보통 다음 구조:

  * `/Page` 객체 안에 `/XObject`로 이미지 포함 (JPEG 등)
  * `/Filter /DCTDecode` → JPEG 스트림
* 텍스트/벡터 없이 단일 이미지 렌더만 수행됨

### ✅ 실현 가능한 접근 방식

#### 1. 클라이언트 측 구현 (Zotero 플러그인)

* `pdf-lib`를 사용하여 XObject 이미지 stream 추출
* 브라우저가 지원하는 이미지라면 (예: JPEG), `Blob`으로 만들어 `Image` 객체로 사용 가능

```js
const blob = new Blob([imageData], { type: 'image/jpeg' });
const img = new Image();
img.src = URL.createObjectURL(blob);
```

* 제한 사항:

  * JBIG2, CCITT 압축은 브라우저 디코딩 불가
  * sandbox iframe(CSP, cross-origin, WebAssembly 제한)
  * Zotero PDF 뷰어 내부에서는 실행 불가

#### 2. 서버 측 구현 (권장)

* 플러그인에서 PDF 바이너리를 서버로 전송 (또는 PDF path 전달)
* 서버에서 다음 수행:

  * PDF 분석 → 스캔 여부 판단
  * 이미지 추출 (pdf-lib, PyMuPDF 등)
  * OCR 적용 (Tesseract, TrOCR 등)
  * LLM을 통해 bibliographic metadata 추출

### 💡 향후 통합 전략

| 컴포넌트                 | 역할                             |
| -------------------- | ------------------------------ |
| `RefSense Plugin`    | PDF 선택, 서버에 처리 요청, 사용자에게 결과 표시 |
| `PDFInspector` (서버)  | PDF 내부 구조 분석, 이미지 추출           |
| `OCRPipeline` (서버)   | 이미지 텍스트화 처리                    |
| `MetaExtractor` (서버) | OCR 텍스트로부터 서지 정보 추출 (LLM 기반)   |

### 🧠 메모 요약

> "텍스트가 없는 PDF는 렌더링보다는 이미지 추출 + OCR로 충분히 처리 가능하다. 브라우저 환경에서는 JPEG 디코딩만으로 미리보기나 분석 기반을 제공할 수 있으며, 전체 분석은 서버 측으로 위임하는 것이 구조적으로 바람직하다."

## 🛠️ 번들링 도구 (Rollup) 도입 검토

### 🎯 **현재 결론: 번들링 도입하지 않음**

RefSense 플러그인의 현재 개발 단계에서는 **모듈 구조 유지**가 가장 합리적이고 안정적인 선택입니다.

### ✅ **현재 방식 평가**

| 측면 | 현재 방식 평가 |
|------|----------------|
| **Zotero 7 호환성** | Services.scriptloader.loadSubScript()가 기본 전제이므로, 별도 JS 파일 유지가 자연스러움 |
| **디버깅/개발 편의성** | 변경 후 Zotero 재시작 없이 개별 파일 교체만으로 테스트 가능 |
| **파일 크기** | 25KB 정도로, 번들링의 성능 이점은 미미함 |
| **로딩 성능** | Zotero 자체가 JS 로더를 최적화하지 않기 때문에 다중 모듈도 큰 부담 없음 |
| **의존성 복잡도** | 현재까지는 내부 모듈만 사용 중이므로 rollup 도입 필요성 낮음 |

### 🔮 **Rollup 도입 고려 시점**

| 시점 | 고려 이유 |
|------|-----------|
| **플러그인 기능이 대규모로 확장됨** | 모듈 수 30개 이상 또는 서브디렉토리 구조 |
| **외부 라이브러리 도입** | 예: lodash, marked, pdf-lib 등 |
| **배포용 난독화/압축 필요** | 보안성, 배포 최적화를 위한 minify, uglify |
| **번역, 설정, 리소스 파일 등 동적 처리 필요** | 리소스를 JS로 변환해 함께 번들링할 경우 |

### 🏗️ **현재 구조에서의 권장 정리 방식**

- **bootstrap.js**: 초기화 담당
- **modules/*.js**: 책임 기반으로 나눔
- **RefSense.ModuleName** 네임스페이스로 연결
- **loadSubScript** 사용 시 순서 제어만 유의

### 📋 **추가 모듈 개선 계획**

현재 모듈 구조를 유지하면서 생산성과 안정성을 높이기 위한 개선사항:

| 제안 | 이유 | 상태 |
|------|------|------|
| **modules/logger.js** | 모든 모듈에서 동일한 RefSense.log() 인터페이스 사용 | ✅ **완료** |
| **modules/commands.js 분리** | 메뉴, 단축키, 명령 등록 등을 한 곳에서 관리 | 📋 **계획 중** |
| **modules/state.js 사용** | 설정값 등을 메모리 캐시에 유지 (성능 ↑) | 📋 **계획 중** |
| **modules/test.js 따로 두기** | 개발 중만 로드되는 테스트 도구 (비공식) | 📋 **계획 중** |

### 🎯 **최종 결론**

**Zotero 7의 특성과 현재 플러그인의 복잡도를 고려하면, 번들링보다는 모듈 구조 유지가 더 적합합니다.**

→ 지금 방식으로 계속 개발하면서, 나중에 필요성이 생길 때 rollup 도입을 고려하면 됩니다.  
→ 필요하면 minify나 자동 .xpi 생성만 스크립트에 추가해도 충분히 배포 최적화가 가능합니다.
